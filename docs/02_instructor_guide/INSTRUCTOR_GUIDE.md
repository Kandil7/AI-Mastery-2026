## 🎓 Teaching Philosophy

### White-Box Approach

**Core Principle:** Students must understand the "why" before the "how"

**Teaching Sequence:**
1. **Math First** - Derive on paper
2. **Code Second** - Implement from scratch
3. **Libraries Third** - Use production libraries
4. **Production Always** - Deploy and monitor

### Active Learning Strategies

1. **Think-Pair-Share**
   - Students think individually (2 min)
   - Discuss with partner (3 min)
   - Share with class (5 min)

2. **Live Coding**
   - Instructor codes live
   - Students follow along
   - Mistakes are learning opportunities

3. **Peer Instruction**
   - Students teach each other
   - Code reviews
   - Pair programming

4. **Project-Based Learning**
   - Real-world projects
   - Portfolio-worthy work
   - Industry-relevant skills

## 💻 Teaching Best Practices

### Code Teaching

**Do:**
- ✅ Code live (don't just show slides)
- ✅ Make mistakes intentionally (show debugging)
- ✅ Explain each line of code
- ✅ Use meaningful variable names
- ✅ Add comments as you code
- ✅ Test code frequently

**Don't:**
- ❌ Copy-paste large blocks
- ❌ Skip error handling
- ❌ Assume prior knowledge
- ❌ Rush through complex code
- ❌ Ignore student questions

### Math Teaching

**Do:**
- ✅ Start with intuition
- ✅ Use visualizations
- ✅ Connect to code
- ✅ Show step-by-step derivations
- ✅ Provide practice problems

**Don't:**
- ❌ Jump into formulas
- ❌ Skip assumptions
- ❌ Ignore geometric interpretation
- ❌ Rush through proofs

### Lab Facilitation

**Do:**
- ✅ Provide starter code
- ✅ Give clear instructions
- ✅ Walk around and help
- ✅ Encourage experimentation
- ✅ Review solutions together

**Don't:**
- ✅ Give answers immediately
- ❌ Do the work for students
- ❌ Ignore struggling students
- ❌ Skip code review

## 🎯 Assessment Guidelines

### Quiz Administration

**Before Quiz:**
- Announce quiz 1 week in advance
- Provide study guide
- Hold review session

**During Quiz:**
- Clear instructions
- Time limits enforced
- No external resources (unless open-book)

**After Quiz:**
- Review answers in class
- Provide detailed solutions
- Allow re-take for partial credit

### Project Evaluation

**Rubric Template:**

| Criteria | Excellent (90-100%) | Good (70-89%) | Satisfactory (60-69%) | Poor (<60%) |
|----------|---------------------|---------------|----------------------|-------------|
| **Functionality** | All features work perfectly | Most features work | Basic features work | Major issues |
| **Code Quality** | Clean, documented, tested | Minor issues | Some issues | Poor structure |
| **Understanding** | Deep explanations | Good explanations | Basic explanations | Unclear |
| **Innovation** | Highly creative | Some creativity | Standard approach | No creativity |
| **Presentation** | Professional, clear | Good presentation | Adequate | Poor presentation |

### Grading Scale

| Percentage | Letter | Description |
|------------|--------|-------------|
| 90-100% | A | Excellent - Exceeds expectations |
| 80-89% | B | Good - Meets expectations |
| 70-79% | C | Satisfactory - Approaching expectations |
| 60-69% | D | Poor - Below expectations |
| 0-59% | F | Fail - Does not meet minimum standards |

## 🗣️ Communication Strategies

### Explaining Complex Concepts

**Framework: ELI5 (Explain Like I'm 5)**

1. **Simple Analogy**
   - "Think of a neural network like a team of experts..."

2. **Visual Representation**
   - Show diagram
   - Walk through step-by-step

3. **Code Example**
   - Show implementation
   - Run and demonstrate

4. **Real-World Application**
   - "This is how Netflix recommends movies..."

### Handling Questions

**Question Types & Responses:**

1. **Clarification Questions**
   - "What does this line do?"
   - → Answer directly

2. **Conceptual Questions**
   - "Why do we use gradient descent?"
   - → Explain intuition, then math

3. **Application Questions**
   - "How would this work for images?"
   - → Discuss, show example

4. **Advanced Questions**
   - "What if we use a different optimizer?"
   - → Acknowledge, explore together, or defer

### Giving Feedback

**Positive Feedback:**
- "Great job implementing the neural network!"
- "Your code is very clean and well-documented."

**Constructive Feedback:**
- "Your implementation works, but consider adding error handling."
- "Good start! Let's optimize this loop for better performance."

**Correcting Mistakes:**
- "I see what you're trying to do. Let's think about..."
- "That's a common misconception. Actually..."

## 📊 Tracking Student Progress

### Key Metrics to Monitor

| Metric | Target | Action if Below Target |
|--------|--------|----------------------|
| Attendance | 80%+ | Reach out to absent students |
| Quiz Scores | 80%+ | Offer review sessions |
| Lab Completion | 90%+ | Provide extra help sessions |
| Engagement | Active in Discord | Personal check-in |
| Project Quality | Rubric-based | 1-on-1 mentoring |

### Early Warning Signs

**Students at Risk:**
- Missing multiple sessions
- Low quiz scores (<60%)
- No lab submissions
- Not active in community
- Expressing frustration

**Intervention Strategies:**
1. Personal email/call
2. Offer 1-on-1 session
3. Pair with strong student
4. Adjust pace if needed
5. Provide additional resources

## 🐛 Common Student Issues

### Technical Issues

**Issue:** "Python not found"
**Solution:** Check PATH, reinstall Python

**Issue:** "Module not found"
**Solution:** Activate venv, pip install

**Issue:** "Code runs slow"
**Solution:** Check hardware, use Colab

**Issue:** "Git errors"
**Solution:** Re-clone, check credentials

### Learning Issues

**Issue:** "Math is too hard"
**Solution:** Provide additional resources, 1-on-1 help

**Issue:** "Can't keep up with pace"
**Solution:** Offer recorded sessions, extend deadlines

**Issue:** "Don't understand code"
**Solution:** More live coding, pair programming

**Issue:** "No time for labs"
**Solution:** Time management tips, prioritize
